# 首诊报告人工质检清单

## 基础信息

- 客户编号是否正确。
- 客户名称是否正确。
- 平台是否正确。
- 类目是否正确。
- 数据周期是否正确。

## 数据口径

- 是否有缺失字段。
- 是否有明显异常值。
- 支付金额、订单数、客单价是否逻辑一致。
- 访客数、订单数、转化率是否逻辑一致。
- 投放消耗、投产比是否口径一致。

## 数据可用性等级

- 是否先判断数据等级，再决定报告允许输出到什么深度。
- 只有单周基础数据时，是否只输出首诊假设、当前样本观察和低风险验证动作。
- 有近4周连续数据时，是否只输出店铺自身历史对比，不冒充行业/同层级判断。
- 有近12周连续数据时，才输出趋势判断和波动区间。
- 有平台同层级或行业/同层级数据时，才输出行业/同层级判断，并标注来源和口径。
- 同时有毛利率和投放归因时，才输出投放效率判断。
- 同时有商品成本和库存数据时，才输出利润和库存策略建议。
- 是否遵守“没有数据源，就不输出强判断；有数据源，就标注来源和口径”。

## 诊断结论

- 是否把可能原因写成确定原因。
- 是否存在过度承诺。
- 是否有凭空判断。
- 是否对数据缺失做了边界说明。

## 商业安全表达

- 无活动前后对比、历史同期数据、渠道拆分或归因数据时，是否避免使用“证明、已经带动、导致、造成、跑通、验证了”等强因果词。
- 无目标值、历史均值、行业均值或同层级对标时，是否避免使用“未达到预期、低于预期、成交缺口、转化率偏低、访客量达标”等预期判断。
- 无历史均值、行业均值、平台同层级数据或连续多周数据时，是否避免使用“稳定、无异常、健康、合理、优秀、达标、偏低、偏高”等状态判断。
- 无上周、上月、活动前后、历史均值或连续多周数据时，是否避免使用“上涨、下滑、改善、恶化、变好、变差、同步放大、没有跟上”等趋势判断。
- 第 07 条如果声明不输出“达标/合理/优秀/偏低/砍预算/加预算”等强判断，正文前文是否也没有出现同类表达。
- 是否存在未说明来源的固定阈值，例如 60%、70%、80%、30%。如必须保留，是否说明为“本次试点观察线，仅用于初步排查，不代表行业标准”。
- 是否把高风险动作写成了直接执行建议，例如加预算、砍预算、降价、停推新品、大范围改版。若数据不足，应改为“验证后再决定”。

## 行动建议

- 是否具体。
- 是否能在一周内执行。
- 是否有验证指标。
- 是否符合客户平台和类目。
- 是否优先输出低风险验证动作：拉数据、核口径、拆渠道/人群/关键词/商品、看客服咨询和用户反馈、制作优化方案。
- 涉及页面替换、预算调整、投放定向、价格机制、库存、停推新品或全店改版时，是否写成“小范围测试/具备条件时验证/验证后再决定”，而不是直接全量执行。

## 版本评级

- 内部验收稿：方向可用，但存在明显强判断、动作风险或客户阅读障碍。
- 试点客户版：商业安全边界基本成立，可小范围发真实商家测试反馈。
- 正式客户版：表达安全、客户可读、动作低风险、后续承接自然，适合稳定交付。

## 防反推

- 不出现 Coze、Notion、GitHub Actions、豆包、火山方舟等内部工具链说明。
- 不出现 Prompt 原文。
- 不出现规则阈值公式。
- 不出现内部模块名。

## 交付前最终检查

确认报告能回答：

1. 这家店当前最大问题是什么？
2. 下周先做什么？
3. 为什么建议进入4周经营周报试点？
4. 是否可以进入对外客户版，还是只能作为内部验收稿。
5. 当前版本评级是内部验收稿、试点客户版，还是正式客户版。
